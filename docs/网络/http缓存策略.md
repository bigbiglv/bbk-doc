# [`http` 缓存](https://developer.mozilla.org/zh-CN/docs/Web/HTTP/Guides/Caching)

## 缓存策略

- 通常为 `get` 请求
- 涉及参数
  - `Date`: 响应头携带的服务的响应请求时的时间
- 可分为强缓存和协商缓存, 但是非官方的分类, `RFC9111` 文档 和 `MDN` 文档都没提到


| HTTP 版本 | 强缓存实现方案 | 协商缓存实现方案 | 特点/备注 |
| :--- | :--- | :--- | :--- |
| `HTTP/1.0` | `Expires` | `Last-Modified` / `If-Modified-Since` | 方式较简陋，**依赖本地时间容易出错** |
| `HTTP/1.1` | `Cache-Control` | `ETag` / `If-None-Match` | **目前的主流方式**，优先使用并兼容 1.0 |

### 请求对比依据

- 主要是请求路径， `body` 参数不影响
- 请求头默认情况下也不加入缓存的判断依据, 避免不同环境下的细微变化导致缓存不可用, 可以使用 `Vary` 来指定需要加入判断的请求头

#### `Vary`

- `Vary` 是一个响应头参数, 用于告诉浏览器哪些请求头会影响缓存
- 常见的 `Vary` 值有 `Accept` 、 `Accept-Language` 和 `Accept-Encoding`
- 例如: `Accept-Language` 在不同语言的时候不应该使用同一份缓存数据
  - 响应头携带 `Vary: Accept-Language` , 那浏览器就会根据 `url +  Accept-Language 的值` 来组合判断缓存

### 强缓存 (`Freshness`)

- `http1.0` 时期使用的 `Expires` 和 当前(`1.1`后)主流使用的 `Cache-Control`
- 启发式缓存目前主要是在缺少 `Cache-Control` 和 `Expires` 时的兜底方案, 缓存全靠猜
- 一旦缓存命中，浏览器就不会再发起请求，所以服务端无法控制浏览器不使用新鲜的缓存

#### 配置

##### `http 1.1` (推荐)
- `Cache-Control` 一般是响应头但是也可以是请求头, **作为请求头时一般用于覆盖响应头的缓存策略**
- `Cache-Control`: 响应头参数, **多个相同的参数取第一个有效的**
  - 有效期 (相对时间)
    - `max-age`(秒): 设置资源的最大有效期（相对时间），如 `Cache-Control: max-age=3600`，表示资源在 `3600` 秒（1小时）内有效
    - `s-maxage`(秒): 在公共缓存(CDN)的时间(优先于 `max-age`), 私有缓存(浏览器)忽略
  - 其他配置
    - `public`: 表示资源可以被任何缓存（私有或共享缓存）进行缓存, 主要用于在携带了 `Authorized` 请求头时共享缓存需要缓存的场景
    - `private`: 表示资源只能被私有缓存（仅浏览器）缓存，通常用于用户敏感数据
    - `immutable`: 资源在新鲜期内不发起验证(协商缓存), 即使没配置现在浏览器也会做优化, 针对子资源会直接使用强缓存
    - `no-cache`: 可缓存, 但是使用前必须跑一次协商缓存
    - `no-store`: 
      - 完全禁止缓存，每次请求都从服务器获取, **不建议随意使用**
      - 会失去浏览器前进后退的缓存效果
      - 建议使用 `no-cache` 和 `private` 的组合
      - 无法阻止旧缓存被命中, 因为命中旧缓存时可能发起的请求没有到达服务器, 因此新的 `no-store` 也无法到达

:::tip
`QPACK` 静态表有**报头压缩优化** , 需要其他配置可另起一个 `Cache-Control` 用两行配置
```
36 cache-control max-age=0
37 cache-control max-age=604800
38 cache-control max-age=2592000
39 cache-control no-cache
40 cache-control no-store
41 cache-control public, max-age=31536000
```
:::

##### `http 1.0` 

- `Expires`: 响应头参数, 资源过期的绝对时间, 由于依赖本地时间目前只做兼容存在

#### 缓存判定 (新鲜度 `Freshness Lifetime`)

- 新鲜度: 缓存寿命
  - 1.1: 直接就是 `新鲜度 = max-age`
  - 1.0: `新鲜度 = Expires - Date`
  - 启发式缓存: `新鲜度 = (Date - Last-Modified) * 10%`

- 年龄
  - 涉及参数: 
    - `current_time` (浏览器内部维护): 请求发出的客户端本地时间
    - `Date` (响应头): 服务器生成响应的时间
    - `response_time` (浏览器内部维护): 请求接收到的客户端本地时间, 缓存失效后重新请求时才更新
  - 表现年龄: 资源在路上的时间
    - `response_time - Date`
  - 校正: 资源可能在共享服务器缓存过, 响应头可能会携带 `Age` 代表在缓存服务器的时间, 这里和表现年龄对比取较大的值
  - 资源在本地缓存了多长时间: `Now - response_time`
  - 最终得到: `年龄 = 校正后的年龄 + 资源在本地的时间`

- 是否新鲜: `是否新鲜 = 年龄 < 新鲜度`

#### 缓存位置

- 分为 **磁盘** 和 **内存**
- 同一个资源是有可能在这两处同时缓存的
- 请求的状态码都是 `200` 后标识 `from disk cache` 或 `from memory cache` , 不会发起真实请求
- 缓存读取的顺序是 `Service Worker` > 内存 > 磁盘

| 缓存类型 | 存储内容 | 特点/场景 | 生命周期 | 隐私模式 |
| :--- | :--- | :--- | :--- | :--- |
| **磁盘缓存** (`from disk cache`) | 资源的原始二进制数据 (体积大) | 容量大，用于长期存储 | 长期有效，失效或清理时移除 | 不使用 |
| **内存缓存** (`from memory cache`) | 编译后的数据 (如位图、字节码) | 容量小，速度快；存储频繁访问的资源 (JS/CSS) | 会话级，关闭标签页或内存充足时清理 | 使用 |



#### 启发式缓存

- 应该配置好 `Cache-Control` 避免使用这种启发式缓存
- 在 `Cache-Control` 还没有被广泛支持前的方案, 缓存靠猜
- 目前在没有 `Cache-Control` 和 `Expires` 的时候作为兜底方案, 属于强缓存
- 主要依赖 `Date` 和 `Last-Modified`
- 现在应该优先使用 `Cache-Control`
- 越久没被修改的文件则缓存的时间就越长
- 缓存到期后走协商缓存
- 缓存判断: 首次响应的时候就通过公式计算缓存寿命
  - 公式: `缓存寿命 = (Date - Last-Modified) * 10%`
  - 通过公式得到缓存寿命以及缓存到达时间, 后续请求做新鲜度对比
    - 当前年龄: `当前年龄 = 当前时间 - 资源到达时间`
    - 新鲜: `当前年龄 < 缓存寿命`
    - 不新鲜: `当前年龄 >= 缓存寿命` , 发起协商缓存进入下一个周期计算 
- 一些极端情况
  - 没有 `Date`: 按照 `HTTP/1.1` 规范（RFC 9110）, 强制要求携带, 即使没有浏览器也会使用本地时间作为 `Date` 值
  - 没有 `Last-Modified` 但是有 `ETag`, 缓存后寿命记为0, 依赖 `ETag` 后续请求走协商缓存判断是否使用缓存
  - 没有 `Last-Modified` 和 `ETag`, 依旧进行本地缓存，但后续都会请求最新数据

### 协商缓存 (`Validation`)

- `http1.0` 时期使用的 `Last-Modified` 文件修改时间 和 当前(`1.1`后)主流使用的 `ETag` 文件指纹
- 强缓存失效或者 `Cache-Control: no-cache` 时触发
- 响应头有带 `Etag` 或者 `Last-Modified`的时候就能走协商缓存, 即使没有配置 `Cache-Control`
- 一定会发起请求和服务器创建链接，由服务器决定返回数据还是客户端使用缓存
- 返回`304`则使用客户端缓存，`200` 则返回新数据
- `RFC9110` 建议服务器在 `200` 响应中尽可能同时发送 `ETag` 和 `Last-Modified` 头, 验证器优先使用 `If-None-Match`

##### `http 1.1` (优先)

- 使用文件指纹的方式, 更精准, 对服务器有一定压力, 浏览器优先使用
- 优点: 根据文件哈希值判断内容精确度高，支持并发修改检测，对于经常变动的资源也能很好的支持，对于大文件的改动即使改动小也能察觉
- 缺点: 服务计算哈希值有性能开销，哈希碰撞
- 配置
  - `ETag`: 响应头配置的服务器生成的任意值, 一般是文件指纹或版本号, 精确度高
  - `If-None-Match`: 请求头携带上次拿到 `ETag` 值供服务器对比

##### `http 1.0`

- 优点: 简单易用 性能好 服务器和浏览器广泛支持
- 缺点: 文件修改时间不够精确(秒)，动态生成的资源无法准确反映内容变化, 内容不变文件重新生成也会触发
- 配置
  - `Last-Modified`: 响应头配置的文件最后修改时间, **不单单是缓存使用**还是显示最后修改时间，被爬虫用于调整爬取频率，以及用于其他各种用途
  - `If-Modified-Since`: 请求头携带上次拿到 `Last-Modified` 值供服务器对比

## 缓存位置

私有缓存和共享缓存, 一个是只能存在个人本地客户端的, 一个是用于 `cdn` 等公共服务器的

### 私有缓存

1. 只存在自己电脑的浏览器上, 具有一定的私密性
2. 请求头携带了 `Authorized` 也正常缓存
3. 使用 `private` 可以确保只被私有缓存进行缓存

### 共享缓存

- 共享缓存位于客户端和服务器之间, 储存一些所有用户公共的资源缓存
- 主要是中间代理服务器, `CDN` 代理, `nginx` 代理
- 请求头携带了 `Authorized` 请求头的请求不能在共享缓存中缓存, 需要设置 `public` 响应头才能进行缓存
- 多个重复的请求到达中间服务器时, 会合并为一次发送给源服务器, 服务器返回数据后中间服务器再分发给所有请求

:::tip 对于 `json` 数据的 `api` 请求
大多数都是不设置缓存，`ETag` 可以根据 `json` 内容生成哈希值，`Last-Modified` 可以使用数据库改动时间
:::


:::tip 缓存清理机制
浏览器会根据缓存大小，当前可用空间来清理一些不常用的缓存数据
:::

### 应用和注意

- 现代前端的静态资源一般是
  - `index.html`: `Cache-Control: no-cache` 保证资源每次都通过协商缓存拿到最新
  - 带哈希的文件: `Cache-Control: public, max-age=31536000, immutable` 文件名自带哈希直接无脑缓存
- `ctrl+f5`: 请求头携带 `no-cache` 直接不走缓存, 协商缓存都不走
- `Cache-Control: max-age=0, must-revalidate` 和 `no-cache` 效果一致, 历史遗留的方式现在应该直接使用 `no-cache`

- 浏览器场景
  1. 首次进入: 正常
  2. `F5`: 快速检查
    - 主文档强制协商缓存, 其他资源可能走强缓存
    - 请求头携带 `Cache-Control: max-age=0`
    - 无法跳过 `Service Worker` 的缓存拦截
  3. `Ctrl + F5`: 推倒重来
    - 跳过所有缓存(包括 `cdn` 缓存), 直达服务器
    - 请求头携带 `Cache-Control: no-cache`
    - 可以跳过 `Service Worker` 的缓存拦截
  4. 勾选"停用缓存": 开发者模式
    - 只要开着控制台就会彻底禁用本地存储
    - 请求头携带 `Cache-Control: no-cache`
